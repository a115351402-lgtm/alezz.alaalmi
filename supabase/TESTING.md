# مصفوفة اختبار النظام — العز العالمي

يُنفَّذ هذا السيناريو بعد أي تغيير جوهري على قاعدة البيانات أو الصفحات.
الروابط: الإنتاج `https://alezz-alaalmi.vercel.app` — أي صفحة تُفتح بإضافة المسار (`/auth`, `/account`, `/admin`).

## 1) الزائر (غير مسجل)

| الاختبار | النتيجة المتوقعة |
|---|---|
| تصفح الرئيسية والسيارات والمزادات | تعمل كما قبل النظام تماماً + زر «تسجيل الدخول» في الشريط |
| الضغط على قلب ♡ في بطاقة | نافذة «سجّل الدخول لحفظ المفضلة» |
| زر «اطلب هذه السيارة» في التفاصيل | نافذة: تسجيل الدخول / المتابعة عبر واتساب |
| زر واتساب | يعمل كما هو دائماً |
| فتح `/account` أو `/admin` مباشرة | تحويل فوري لصفحة الدخول / الرئيسية |
| قراءة جدول vehicles عبر REST بمفتاح publishable | يرى صفوف `approved` فقط |

## 2) العميل (customer)

| الاختبار | النتيجة المتوقعة |
|---|---|
| تسجيل حساب جديد (بريد + جوال) | دخول فوري، صف في profiles بدور customer تلقائياً |
| تسجيل برقم جوال مستخدم | رسالة عربية واضحة قبل الإرسال |
| قلب ♡ على سيارة | ❤ + تظهر في `/account#favorites` |
| «اطلب هذه السيارة» (Encar ومزاد) | طلب pending_payment بسعر السيرفر + مرحلة أولى + إشعار |
| الطلب مرة ثانية لنفس السيارة | «لديك طلب قائم» — لا تكرار |
| تغيير الأدمن لحالة الطلب | تحديث حي بدون refresh + إشعار تلقائي |
| فتح `/admin` | تحويل للرئيسية |
| من console: `update user_roles` / `insert orders` / قراءة طلبات الغير | **رفض RLS في كل الحالات** |
| تعديل إشعار (غير is_read) من console | رفض — العمود الوحيد المسموح is_read |

## 3) اللوجستي (logistics)

| الاختبار | النتيجة المتوقعة |
|---|---|
| فتح `/admin` | يرى تبويب **الطلبات فقط** (بدون قائمة تغيير الحالة وبدون بيانات العملاء الشخصية) |
| إضافة مرحلة شحن | تُحفظ وتظهر **حية** عند العميل + toast |
| من console: تعديل vehicles / قراءة profiles / user_roles | رفض RLS |

## 4) الأدمن (admin)

| الاختبار | النتيجة المتوقعة |
|---|---|
| استيراد سيارة من Encar/مزاد بمعرّفها | تظهر pending بصور وسعر السيرفر |
| اعتماد سيارة | تظهر فوراً للزوار (approved) |
| سيارة يدوية + رفع صور وتقرير | روابط storage عامة تعمل |
| تغيير حالة طلب | إشعار تلقائي للعميل (تريغر) |
| طلب يدوي لعميل | يظهر في حساب العميل |
| إنشاء حساب عميل | يعمل عبر `/api/admin` |
| إنشاء حساب **موظف** أو الإدراج المباشر في user_roles | **403 / رفض RLS** — حكر على super_admin |

## 5) المالك (super_admin)

| الاختبار | النتيجة المتوقعة |
|---|---|
| كل صلاحيات الأدمن | تعمل |
| تبويب «فريق العمل» | ظاهر — إنشاء admin/logistics ومنح/سحب الأدوار |
| سحب دور موظف | يفقد الوصول فوراً (يُحوَّل خارج اللوحة عند التحديث) |

## 6) فحوص خلفية (عبر أدوات سوبابيس)

- `get_advisors(security)` → فقط الملاحظات الخمس الموثقة (phone_taken لـ anon + دوال الأدوار لـ authenticated) + ملاحظة Leaked-Password (ميزة Pro مؤجلة).
- `get_advisors(performance)` → لا تحذيرات (ملاحظات unused_index طبيعية).
- التريغرات: `on_auth_user_created` على auth.users، `trg_orders_notify_status` على orders، `trg_profiles_updated_at` على profiles.
- publication `supabase_realtime` تشمل orders + order_milestones + in_app_notifications.

## مؤجلات معلومة (بقرار المالك)

- مزود SMTP خاص + إعادة تفعيل تأكيد البريد.
- حماية كلمات المرور المسربة (تتطلب خطة Supabase Pro).
- حالة «إلغاء الطلب» + الدفع الإلكتروني.
- تقييد أعمدة orders عن اللوجستي على مستوى قاعدة البيانات (مقيد واجهياً فقط حالياً).
